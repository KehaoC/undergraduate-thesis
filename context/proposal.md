# 毕业论文 Proposal

## 题目

**SkillRank：面向大规模 Agent Skill 价值发现的云端系统设计与实现 **

---

## 一、时代背景

互联网诞生之初，人类面临的问题是：**网页太多，怎么找到有价值的那一个？**

Google 的答案是 PageRank——不靠中心化编辑，靠网页之间天然产生的引用关系来判断价值。这个洞察改变了世界。

现在，Agent 时代正在重演同样的问题：**Skill 太多，怎么让 Agent 找到有价值的那一个？**

ClawHub 上已有数万个 Skill，还在持续增长。每一个 Skill 都声称自己有用。但 Agent 没有判断力——它只能在上下文窗口里看到被装入的 Skill，无法知道哪个更值得信任。

---

## 二、问题：Skill 越多，Agent 越蠢

这不是直觉，是已被学术研究量化的事实。

**SkillsBench（OSU / Stanford / CMU, arxiv:2602.12670）** 对 86 个任务、7000+ 条测试的系统评估发现：

- Skill 效果是**倒 U 型曲线**：2~3 个 Skill 时效果最佳，超过 4 个之后提升骤降
- 有 **1/5 的任务**在装了 Skill 之后性能反而下降，最大降幅 39.3pp
- 过于详细的 Skill 文档反而让性能下降 2.9pp

**Single-Agent with Skills（加拿大 / 韩国, arxiv:2601.04748）** 测试了不同 Skill 库规模下的路由准确率：

- Skill 数量 < 20：选择准确率 > 90%
- Skill 数量 > 30：准确率开始陡降
- Skill 数量 = 200：准确率跌至约 **20%**

这种崩溃不是线性的，在 20~30 个附近存在临界点，越过之后准确率呈悬崖式下滑。

**根本原因**：当大量 Skill 同时出现在上下文中时，语义相似的 Skill 让模型无法区分，选择失败率随数量非线性飙升。同时研究也验证，将 Skill 分层组织后，准确率可以从 45% 恢复到 85%。

---

## 三、为什么现有解法不够

面对这个问题，当前社区的主流建议是：**用户手动管理**。

控制数量（装 8~15 个）、规范命名、分组分类、定期清理……

这是在让用户承担系统应该解决的问题。

更根本的缺陷是：**手动管理无法扩展，也无法产生价值积累。**

- 用户 A 花了一周整理出最优 Skill 组合，这个知识只在他自己的本地
- 用户 B 面对同样的问题，只能重新来过
- 数万用户的试错经验，没有任何一种机制将它汇聚起来

互联网曾经也面临同样的困境。早期的 Yahoo! 目录靠人工编辑分类网页——可以工作，但无法扩展。PageRank 的突破在于发现了一个**分布式的、自动产生的质量信号**：超链接。每一个创作行为都在无意间为整个互联网的质量评估做出贡献。

**Skill 世界缺少的，正是这样一个信号。**

---

## 四、核心问题

> **在 Agent 完成真实任务的过程中，是否存在一种自然产生的信号，能够反映 Skill 的真实价值——就像超链接之于网页？**

这个问题有三层递进：

**第一层：信号存在吗？**

当用户调用 Agent 完成任务时，产生了大量隐含信号：哪些 Skill 被调用了、调用顺序是什么、任务最终完成了吗、用户有没有重试、用户在结果上继续构建了什么。这些信号是否足以反映 Skill 的质量？

**第二层：信号如何结构化？**

不同 Skill 之间存在天然的协作关系——完成某类任务，Skill A 之后往往调用 Skill B。这种共现关系能否构成一张图，就像超链接构成了网页图？图上的传播算法（类 PageRank）能否在这张图上工作？

**第三层：这个排序系统的价值在哪里？**

如果 Skill 价值可以被量化和排序，那么：Agent 可以动态选择当前任务最合适的 Skill 子集（而不是把所有 Skill 塞入上下文）；Skill 创作者有了真实的质量反馈；平台有了不依赖人工编辑的质量保证机制。

---

## 五、为什么值得解决

### 5.1 规模上的必然性

ClawHub 的 Skill 数量只会增长，不会减少。手动管理是一个在规模面前必然失效的方案。这个问题不是"如果"，是"什么时候"。当 Skill 数量从数万增长到数十万，当每个用户的 Agent 可以接入全库，一个能自动判断 Skill 价值的系统就从"有用"变成"必须存在"。

### 5.2 问题的新颖性

Skill 路由不是传统的信息检索问题，也不是传统的推荐系统问题。

传统搜索：用户主动查询，找到**信息**，阅读后判断价值。
传统推荐：根据用户历史，推荐**内容**，用户消费后产生反馈。
Skill 路由：Agent 在执行**任务**过程中，需要实时选择**能力**，价值体现在任务完成质量上。

"能力"不是"信息"，也不是"内容"。能力的价值只在被使用时才得以验证。这是一个新的问题空间，现有方法不能直接套用。

### 5.3 数据壁垒的形成

如果这个系统能够建立起来，它会产生一个强数据壁垒：

```
用户越多 → 任务完成信号越多 → Skill 价值图越准确
→ Agent 越聪明 → 用户越多
```

PageRank 的壁垒不是算法（算法可以被复制），是**数据**——Google 积累的点击行为和链接图谱。同样，Skill Rank 的壁垒不是排序公式，是任务完成数据。这是先发者的核心优势。

---

## 六、本论文的定位

本论文不试图解决全部问题，而是聚焦在**问题的定义和核心机制的验证**：

1. **形式化定义** Skill 价值发现问题：什么是好的 Skill 排序？如何评估？
2. **识别和采集** 任务完成过程中的隐含质量信号
3. **构建** Skill 共现图，验证图结构是否能反映 Skill 价值
4. **实现** 一个 production-ready 的路由系统，接入 ClawHub 全库，在真实用户任务中验证

最终交付一个可以上线、可以持续从用户行为中学习的系统——而不是一个实验室里的原型。
