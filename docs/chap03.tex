\chapter{Taste 系统总体设计}
\label{cha:design}

\section{系统架构概述}
\label{sec:arch_overview}

Taste 系统的设计核心目标是：将 Agent Skill 的发现、质量评估与路由能力从用户本地彻底迁移至云端，构建一个对用户完全透明、对 Agent 调用友好的 Skill 能力引擎。正如 Google 通过将网页索引迁移至服务器端彻底改变了信息检索的方式，Taste 通过将 Skill 索引与质量评估迁移至云端，使得 Agent 无需在本地维护任何 Skill 列表，也无需承担意图匹配的计算压力。

系统整体采用前后端分离的分层云端架构，如图 \ref{fig:system_arch} 所示，从下至上分为四层：数据采集与存储层、算法计算层、服务接口层和前端交互层。各层之间通过标准化接口松散耦合，每一层可以独立扩展而不影响其他层的运行。

系统的主数据流向如下：数据采集层持续从 ClawHub 平台爬取 Skill 原始数据，写入 PostgreSQL；算法计算层作为后台批处理任务（由 Celery 调度），定期对全量 Skill 执行 Content embedding、稀疏 K-NN 图构建、PageRank 迭代、HDBSCAN 语义聚类和 UMAP 降维，将计算结果回写数据库；服务接口层基于 FastAPI 实现，在用户提交查询时实时运行 Description embedding 和 FAISS 检索，返回推荐结果；前端层以三维地形图为核心可视化组件，提供直观的交互式 Skill 探索界面。Redis 作为缓存层贯穿服务接口层，将高频查询的推荐结果和 Description embedding 向量缓存至内存，将 API 查询的 P99 延迟控制在 500ms 以内。

\section{数据接入与管理模块设计}
\label{sec:data_module}

数据接入模块负责从外部平台持续获取 Skill 数据并维护其新鲜度。系统目前接入 ClawHub 作为主要数据源，ClawHub 是目前规模最大的 Agent Skill 社区平台，收录了来自全球开发者贡献的 2 万余个 Skill，覆盖面广且数据质量相对可控。

数据接入流程分为初始全量拉取和增量同步两个模式。初始全量拉取在系统首次部署时运行，以异步并发的方式遍历 ClawHub 的全部 Skill 列表页，对每个 Skill 访问其详情页以提取完整的 Description 和 Content 字段，同时对含 GitHub 仓库链接的 Skill 调用 GitHub REST API 异步获取 Star 数和 Fork 数。增量同步每 24 小时运行一次，仅抓取自上次同步以来新增或更新的 Skill，维护数据的时效性。

数据清洗分三个阶段执行：格式规范化（去除冗余空白、HTML 残余、控制字符）、去重过滤（MD5 精确去重 + embedding 相似度近重复检测，阈值 0.98）、有效性过滤（剔除字段为空或内容无意义的条目）。原始爬取数据的有效率约为 64.9\%（2103/3241）。

\section{SkillRank 计算模块设计}
\label{sec:skillrank_module}

SkillRank 计算模块是 Taste 系统的算法核心，完整执行流程如图 \ref{fig:skillrank_pipeline} 所示，分为四个依次进行的阶段：Content Embedding、稀疏 K-NN 图构建、PageRank 迭代收敛、语义聚类与 UMAP 降维。

\centering
\begin{tikzpicture}[

  note/.style={font=\footnotesize, text width=2.8cm, align=center}
]
  \node[stage] (S1) at (0,0) {Content\\Embedding};
  \node[stage] (S2) at (3.5,0) {FAISS\\K-NN 图构建};
  \node[stage] (S3) at (7,0) {PageRank\\幂迭代};
  \node[stage] (S4) at (10.5,0) {聚类 \&\\UMAP 降维};

  \draw[arrow] (S1) -- (S2);
  \draw[arrow] (S2) -- (S3);
  \draw[arrow] (S3) -- (S4);

  \node[note] at (0,-1.5) {sentence-transformers\\all-MiniLM-L6-v2\\384 维向量};

  \draw[dashed] (10.5,-0.8) -- (10.5,-1.0);
\end{tikzpicture}
\caption{SkillRank 四阶段计算流程。从左至右依次执行 Content 向量编码、稀疏亲缘图构建、PageRank 迭代求解、语义聚类与可视化降维。}
\label{fig:skillrank_pipeline}
\end{figure}

第一阶段使用 \texttt{all-MiniLM-L6-v2} 模型对所有 Skill 的 Content 和 Description 字段分别进行编码，输出 384 维稠密向量。该模型在 STS（语义文本相似度）基准任务上表现稳定，CPU 环境下推理速度约为每秒 1000 条文本，适合大批量 offline 编码。

第二阶段利用 FAISS IVF 索引为每个 Skill 检索 $K=15$ 个 Content 最近邻，构建稀疏有向亲缘图。图以压缩稀疏行（CSR）格式存储，每条边携带余弦相似度作为传播权重。

第三阶段在 CSR 格式的稀疏图上运行幂迭代，每轮以 \texttt{scipy.sparse} 矩阵-向量乘法实现，时间复杂度严格为 $O(nK)$，约 20 轮迭代至 $L_1$ 收敛阈值 $10^{-6}$。

第四阶段使用 HDBSCAN 对所有 Skill 的 Description embedding 执行密度聚类（自动识别语义主题），再用 UMAP 将 Description embedding 降至二维，与 SkillRank 分共同构成三维地形坐标 $(x, y, z)$，存入 \texttt{skill\_scores} 表，供前端读取渲染。

\label{fig:query_flow}
\end{figure}

查询流程有意使用 \textbf{Description} embedding 空间（而非 Content 空间）来匹配用户的 context：用户 context 描述的是"我需要什么样的 Skill"，这在语义上与 Skill 的 Description（"我是做什么的"）对齐，而与 Skill 的 Content（"我具体包含什么内容"）并不直接对应。FAISS 检索返回 $\alpha K$（取 $\alpha=7$）个候选后，在候选集内按预计算的 SkillRank 分降序排列，取前 $K$ 个作为最终推荐。

对于已积累行为信号的 Skill，系统以加权融合方式修正最终得分：

\begin{equation}
    \label{eq:final_score}
    \mathrm{Score}(s_i) = (1 - \lambda) \cdot SR(s_i) + \lambda \cdot B(s_i)
\end{equation}

其中 $B(s_i)$ 为行为分（由点击率、选中率和平均停留时长归一化计算），$\lambda$ 为行为权重，系统冷启动阶段 $\lambda=0$，随行为数据积累线性增长至上限 $\lambda=0.3$。

\section{本章小结}
\label{sec:chap3_summary}

本章从系统整体架构出发，完整阐述了 Taste 的设计方案。系统采用四层架构（数据存储、算法计算、服务接口、前端交互），各层职责清晰，支持独立扩展。数据接入模块通过三阶段清洗保证数据质量；SkillRank 计算模块以四阶段流程完成从原始文本到质量分值的完整计算链路；查询路由模块实现了 Redis 缓存优先、FAISS 检索加速的高效在线推荐；可视化前端将抽象建模结果转化为直观的三维地形交互体验；核心 API 接口面向 Agent 程序化调用设计，极简输入极简输出。第四章将进一步介绍上述各模块的具体实现细节与关键工程决策。
