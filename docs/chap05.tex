\chapter{实验评测与总结}
\label{cha:experiment}

\section{实验设计}
\label{sec:exp_design}

本章实验围绕三个问题展开：第一，SkillRank 是否能够在语义邻域内部给出优于非图结构基线的排序；第二，聚类粒度与参数调优是否会显著改变实验结论；第三，在当前真正执行的 100 和 1000 两个规模上，FAISS 近似检索是否已经带来实际的运行时间收益。

\subsection{实验数据与执行边界}

本文以第四章整理得到的 10028 条 GitHub Skill 数据资产为统一输入来源，但本轮定稿阶段只实际运行了 100 和 1000 两个规模的子集。这样的边界是刻意设置的：目标不是在一晚之内盲目堆大规模，而是在可控时间内完整保留两组真实结果、日志与图表。100 规模用于快速观察算法行为与小样本稳定性，1000 规模作为主实验规模；10000 规模代码路径已经具备，但本次不执行。

\subsection{Ground Truth 与评测指标}

本文仍以 GitHub 仓库 Star 数作为弱监督 proxy，而不是把它当作完美真值。具体评测方式为：先在 Description 语义空间内聚类，再在每个聚类内部比较 SkillRank 排序与 Star 排序的 Spearman 秩相关系数，最后按聚类规模加权平均，得到 overall within-cluster Spearman $\rho$。这一指标的含义是：系统是否能够在“同类 Skill”的比较范围内，把社区热度较高或被生态更广泛接受的 Skill 排到更前的位置。

需要说明的是，当前数据集中的多个 Skill 可能来自同一 GitHub 仓库，因此它们会共享同一个 Star 数。这意味着 Star 只能被视为仓库级热度信号，而不是精确的 Skill 级质量标签。本文后续所有分析都在这一限制下进行。

\section{效率评测结果}
\label{sec:scalability_results}

为了判断 FAISS 在当前已执行规模上是否真的带来时间收益，本文额外对“精确 K-NN 图构建 + PageRank”与“FAISS 近似 K-NN 图构建 + PageRank”做了同配置对比。结果见表 \ref{tab:runtime_real}。

\begin{table}[htbp]
\centering
\caption{精确 K-NN 与 FAISS 近似 K-NN 的运行时间对比}
\label{tab:runtime_real}
\begin{tabular}{lrrrr}
\toprule
规模 & 精确检索时间 / s & FAISS 时间 / s & 精确/FAISS & $\rho_{\text{exact}} - \rho_{\text{faiss}}$ \\
\midrule
100 & 0.0007 & 0.0015 & 0.47 & 0.0000 \\

图 \ref{fig:runtime_real} 更直观地展示了这一事实：在 100 与 1000 两个规模上，FAISS 的索引训练和调度开销尚未被样本量摊薄，因此其总耗时反而高于精确检索。这一结果与很多“FAISS 一定更快”的经验判断不同，但它是本轮实验中必须诚实保留的结论。

\section{案例分析}
\label{sec:case_study}

为了观察 SkillRank 在实际簇内的排序形态，本文选取 1000 规模实验中最大的语义簇进行案例分析。该簇的自动标签为“\texttt{api / use / integration}”，共包含 93 条 Skill。表 \ref{tab:case_study_real} 展示了其中 SkillRank 排名前 5 和后 3 的条目。

这一案例有两个现象值得注意。第一，排名靠前的条目并不一定来自高 Star 仓库，例如 \texttt{openapi-spec-generation} 与 \texttt{moai-lang-typescript} 的仓库 Star 都很低，但在结构上处于该簇内部更中心的位置。第二，末尾条目并非简单对应最低 Star 数，而是通常同时具备“结构中心性弱”和“与簇原型偏离更大”这两个特征。也就是说，SkillRank 的排序逻辑并不是在复写 Star，而是在尝试从簇内内容结构中找出“被更广泛模仿、复用或靠近主流模式”的 Skill。

当然，这组案例不能被过度解读。由于当前使用的是仓库级 Star 而不是 Skill 级真值，排名靠前但 Star 很低的条目既可能是真正被低估的高质量 Skill，也可能只是出自一个较小众仓库的结构中心节点。因此，案例分析更适合用来说明 SkillRank 的行为模式，而不是直接宣称其已经发现了“绝对优质”的 Skill。

\section{讨论：当前结果的意义与局限}
\label{sec:limitations}

结合上述实验，本文认为当前结果已经足以支持两个结论。其一，SkillRank 的图结构信息确实提供了增量价值，因为调优后的配置在 100 和 1000 两个规模上都优于最强非图结构基线。其二，在当前样本量下，图结构增益是“温和但稳定”的，而不是“压倒性的领先”。

与此同时，本轮实验也清楚暴露出若干局限。

\textbf{第一，Star 只是仓库级 proxy，不是 Skill 级真值。} 同一仓库中的多个 Skill 共用同一个 Star 数，这会引入标签噪声，并削弱 within-cluster 排序评测的分辨率。

\textbf{第二，100 与 1000 规模不足以证明 FAISS 的时间优势。} 当前实验只能说明近似检索几乎不损失排序质量，但不能据此声称“FAISS 已经更快”。如果后续需要支撑大规模效率结论，必须补跑 10000 及以上规模。

\textbf{第三，当前评测仍停留在离线排序层。} 本文尚未在真实 Agent 任务中比较“注入不同 Top-$K$ Skill 集合后最终任务完成质量”的差异，因此还不能把离线 $\rho$ 的提升直接等同于真实任务成功率的提升。

\textbf{第四，聚类粒度会影响结论的强弱。} 更细的聚类有时会得到更高的 $\rho$，但往往伴随过小簇，导致评测不稳定。因此，后续研究需要引入更系统的聚类稳健性分析，而不是只选择单个最优点。

\section{全文总结}
\label{sec:conclusion}

本文围绕 Agent Skill 生态中的价值发现问题，完成了从理论建模、算法实现到系统原型和实验评测的完整闭环。理论上，本文将 Skill 明确建模为 $\langle Description, Content \rangle$ 二元结构，提出条件排序问题定义；算法上，本文实现了融合 Content 原型中心度与图传播分的 SkillRank；工程上，本文构建了 Taste 原型，打通了离线计算、在线重排和三维语义地形 Demo。

实验方面，本文在 10028 条 Skill 的统一数据资产上，实际完成了 100 和 1000 两个规模的完整实验，保留了日志、摘要 JSON、可视化产物与 3D Demo 截图。调优后的 SkillRank 在 100 规模达到 $\rho=0.0600$，在 1000 规模达到 $\rho=0.0562$，均优于最强非图结构基线；同时，实验也诚实地表明，在这两个规模上精确 K-NN 仍快于 FAISS。换言之，本文已经验证了 SkillRank 在当前数据上的有效性与可运行性，但尚未完成对超大规模效率优势的实证闭环。

对一篇本科毕业论文而言，这一结果是有价值的：本文没有停留在概念性讨论，而是交付了真实代码、真实日志、真实 3D Demo 和可复核的实验结果。后续工作若继续推进，应重点补齐三个方向：一是更高规模下的效率实验，二是真实 Agent 任务层面的在线评测，三是更细粒度的 Skill 级监督信号构建。只有把这三部分补齐，SkillRank 才能从“可工作的研究原型”进一步走向“可广泛部署的能力发现基础设施”。
