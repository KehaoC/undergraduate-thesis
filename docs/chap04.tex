\chapter{系统实现}

\section{开发环境与技术栈}
\label{sec:tech_stack}

本系统的开发环境与技术栈如下：

\begin{itemize}
    \item \textbf{前端}：React、TypeScript、Ant Design
    \item \textbf{后端}：Node.js、Express、TypeScript
    \item \textbf{数据库}：PostgreSQL、Redis
    \item \textbf{部署环境}：Docker、云服务器
    \item \textbf{其他关键组件}：Elasticsearch（全文检索）、GitHub API（数据采集）
\end{itemize}

选型理由：React生态成熟，适合构建复杂交互界面；Node.js与前端技术栈统一，便于全栈开发；PostgreSQL支持复杂查询和事务；Redis提供高性能缓存；Elasticsearch提供强大的全文检索能力。

\section{Skill数据接入实现}
\label{sec:data_ingestion_impl}

数据接入流程如下：

\begin{enumerate}
    \item 从GitHub API获取Skill仓库的原始信息，包括基础元数据和统计指标
    \item 对原始字段进行解析和标准化处理，统一数据格式
    \item 将清洗后的数据写入PostgreSQL数据库
    \item 触发指标计算与评分更新任务
\end{enumerate}

对于缺失值和异常值的处理：缺失的描述字段使用README内容补充；异常的统计数据（如负数）标记为无效并记录日志；重复数据通过URL哈希去重。

\section{特征构建与价值评估实现}
\label{sec:feature_evaluation_impl}

\subsection{特征设计}
\label{subsec:feature_design}

系统构建的候选特征包括：

\begin{itemize}
    \item \textbf{热度特征}：GitHub Star数、Fork数、下载量、浏览量
    \item \textbf{活跃度特征}：最近更新时间、提交频率、Issue响应速度
    \item \textbf{内容质量特征}：描述完整性（字符数）、文档丰富度（README长度）、示例数量
    \item \textbf{可复用性特征}：标签规范性、参数化程度、依赖数量
    \item \textbf{生态反馈特征}：被引用次数、收藏数、社区讨论数
    \item \textbf{内容亲缘特征}：基于Content embedding相似度构建的亲缘图中心性得分
\end{itemize}

\subsubsection{内容亲缘特征的计算}

内容亲缘特征的计算分为以下步骤：

\textbf{步骤1：Content向量化}。使用预训练语言模型（如OpenAI text-embedding-3-small）对每个Skill的Content进行编码，获得768维的向量表示。

\textbf{步骤2：相似度矩阵构建}。计算任意两个Skill之间的余弦相似度，构建$N \times N$的相似度矩阵，其中$N$为Skill总数。为降低计算复杂度，仅保留相似度大于阈值$\theta$（如0.7）的边。

\textbf{步骤3：图中心性计算}。在构建的无向加权图上，计算每个节点的度中心性（Degree Centrality）作为基础得分。度中心性定义为：

\begin{equation}
    \label{eq:degree_centrality}
    DC(s_i) = \sum_{j \neq i} \mathbb{I}(sim(s_i, s_j) > \theta)
\end{equation}

其中，$\mathbb{I}(\cdot)$为指示函数。该指标反映了与某个Skill内容相似的Skill数量，间接衡量了该Skill所代表的解法模式的普遍性。

\subsection{评分规则实现}
\label{subsec:scoring_impl}

各指标归一化采用Min-Max标准化方法：

\begin{equation}
    \label{eq:normalization}
    x_{norm} = \frac{x - x_{min}}{x_{max} - x_{min}}
\end{equation}

权重设置基于业务经验与系统目标，当前采用的启发式权重为：$w_1=0.25$（热度），$w_2=0.15$（活跃度），$w_3=0.20$（质量），$w_4=0.10$（可复用性），$w_5=0.10$（生态反馈），$w_6=0.20$（内容亲缘）。

其中，内容亲缘特征权重设置为0.20，与内容质量特征权重相当，反映了本文对Skill生态结构重要性的重视。热度特征权重最高，体现了社区认可度的基础作用。活跃度和生态反馈权重相对较低，作为辅助判断指标。

\subsection{排序结果生成}
\label{subsec:ranking_generation}

排序结果生成流程：首先计算每个Skill的综合分数，然后按分数降序排列。支持多条件筛选后的二次排序，例如先按标签筛选，再按综合分数排序。

为保证结果可解释性，系统在详情页展示各维度得分的具体数值和占比，用户可以了解Skill得分的构成原因。

\section{前后端功能实现}
\label{sec:frontend_backend_impl}

\subsection{Skill列表页实现}
\label{subsec:list_page_impl}

列表页展示字段包括：Skill名称、简短描述、作者、标签、综合评分、Star数、更新时间。

支持的排序方式：按综合分数、按热度、按更新时间、按创建时间。支持的筛选方式：按标签、按作者、按更新时间范围。

\subsection{Skill详情页实现}
\label{subsec:detail_page_impl}

详情页展示内容包括：

\begin{itemize}
    \item 基础信息：名称、完整描述、作者、来源链接、创建时间、更新时间
    \item 关键指标：Star数、Fork数、下载量、被引用次数
    \item 价值评分：综合得分及各维度得分的可视化展示
    \item 使用说明：安装方法、使用示例、依赖说明
\end{itemize}

\subsection{后端服务实现}
\label{subsec:backend_impl}

后端服务主要实现以下功能：

\textbf{查询接口}：提供GET /api/skills接口，支持分页、排序和筛选参数。使用PostgreSQL的索引优化查询性能，对热门查询结果使用Redis缓存。

\textbf{排序逻辑}：在数据库层面完成排序，利用skill\_scores表的索引加速。对于复杂的多条件排序，先在应用层筛选，再在数据库层排序。

\textbf{数据更新任务}：使用定时任务每日更新Skill基础数据，每周重新计算评分。更新过程采用增量更新策略，只处理变化的数据。

\section{系统部署与运行流程}
\label{sec:deployment}

系统采用Docker容器化部署，主要服务包括：前端服务、后端API服务、PostgreSQL数据库、Redis缓存、Elasticsearch检索服务。

服务协作流程：用户请求首先到达前端，前端调用后端API，后端查询Redis缓存，缓存未命中则查询PostgreSQL，检索请求转发至Elasticsearch。数据更新任务独立运行，定期同步数据并更新评分。

\section{本章小结}
\label{sec:impl_summary}

本章围绕SkillRank的具体工程实现展开，介绍了开发环境、数据接入流程、特征构建与评分实现方法，以及系统前后端核心功能的落地方式。通过这些实现工作，SkillRank完成了从Skill数据采集到价值排序展示的完整功能闭环。
